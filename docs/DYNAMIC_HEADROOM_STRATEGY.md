# Dynamic Headroom Strategy

更新时间: 2026-03-21

## 1. 文档目的

这份文档只说明一件事:

在当前补光灯项目里，为什么固定余量控制只能作为过渡版本，后续必须演进到“动态余量控制”。

本文不改代码，只统一控制逻辑，避免后面继续把几个概念混在一起:

- `ADC_VDS` 采到的是什么
- 真正该让 `Q5` 保留多少压差
- “Q5 接近全开”与“Q5 硬顶死”有什么本质区别
- 当前架构下什么能做，什么还不能做

## 1.1 当前代码已采用的版本

截至现在，固件里已经先落地了一个简化动态版本:

- `light_dac_target_permille` 低时，`Vdrain_gnd_target` 约为 `1000mV`
- `light_dac_target_permille = 1000` 时，`Vdrain_gnd_target` 约为 `500mV`
- 中间区间按线性方式过渡
- 正常闭环增加了 `Vout_soft_max = 39000mV`
- `Vout_abs_max = 42000mV` 仍保留为强制关断保护

这不是最终最优值，只是当前可直接上板验证的第一版。

## 2. 当前已经确认的硬件前提

- 主功率级: `MP1907 + Boost`
- 线性恒流级: `TLV2372 + Q5(IXTH30N50L2) + R14(0.1 ohm)`
- `ADC_VDS` 当前按 `Q5 drain-to-gnd` 电压理解
- `ADC_VOUT` 当前按 `VOUT` 母线电压理解
- MCU 当前没有独立闭环采样 `Iled`
- 亮度命令来自 `TIM3 PWM -> RC -> DAC_CUR_CTR`

所以当前系统的控制职责是:

- 模拟恒流环负责“快电流调节”
- Boost 负责“慢速提供刚好够用的母线电压”

## 3. 必须先分清的三个电压

### 3.1 当前采到的量

`Vdrain_gnd = ADC_VDS`

这是 `Q5` 漏极对地电压，不是 `Q5 true Vds`。

### 3.2 采样电阻上的压降

`Vsense = Iled * 0.1 ohm`

这是 `Q5 source-to-gnd` 的主要组成部分。

### 3.3 Q5 真正承受的压降

`Vds_true = Vdrain_gnd - Vsense`

真正决定 `Q5` 是否仍处于可调节区的，是这个量，而不是单独的 `Vdrain_gnd`。

## 4. 为什么固定 `1V` 只能算第一版

当前固定余量版本，本质上是在控:

`Vdrain_gnd_target = 1000mV`

它的优点是:

- 简单
- 容易验证方向
- 能快速把 `Q5` 的无谓耗散从“大余量”压到“小余量”

但它有一个结构性问题:

`Vdrain_gnd` 固定，不代表 `Vds_true` 固定。

因为:

`Vds_true = Vdrain_gnd - Vsense`

当电流变大时，`Vsense` 变大，所以:

- 同样是 `Vdrain_gnd = 1V`
- 大电流时 `Q5 true Vds` 会更小
- 小电流时 `Q5 true Vds` 会更大

这说明固定 `Vdrain_gnd` 目标并不是最终理想控制量。

## 5. 当前讨论后的正确工程目标

当前项目真正想要的，不是让 `Q5` 长期吃掉很大压差。

真正想要的是:

- 低功率时，`Q5` 可以保留略大一点的调节空间，保证恒流环稳定
- 高功率时，`Q5` 尽量接近全开，少吃压差，少发热
- 但不能直接把 `Q5` 硬顶死，否则模拟恒流环会失去调节权

一句话说清:

高功率目标不是“让 Q5 进入开关模式”，而是“让 Q5 尽量接近全开，只保留最小 keep-alive margin”。

这里的 `keep-alive margin` 指的是:

- 运放和线性 MOS 仍然保有一点点调节权
- LED `Vf` 漂移、输入波动、热漂变化出现时，恒流环还能修正

## 6. “接近全开”与“硬全开”不是一回事

### 6.1 可以接受的状态

`Q5` 很接近全开，但还留有极小调节余量。

这种状态下:

- 线性损耗明显下降
- 模拟恒流环仍然活着
- 亮度和电流仍由模拟恒流环兜住

### 6.2 当前架构下不能直接追求的状态

`Q5` 被长期硬顶死，全开不调节。

因为当前系统里:

- Boost 还没有主电流闭环
- `Q5` 一旦彻底失去调节权
- 电流就不再真正受控，而会更多由 `VOUT` 与 LED 自身 `I-V` 曲线决定

所以在当前架构下:

- 可以追求“尽量接近全开”
- 不能把“硬全开”当成正常工作目标

## 7. 动态余量的正确控制思路

当前固定余量控制，控的是:

`Vdrain_gnd_target = constant`

后续更合理的动态余量控制，应该逐步过渡到:

`Vdrain_gnd_target = Vsense_est + Vds_keepalive_target`

其中:

- `Vsense_est` 是对 `Iled * 0.1 ohm` 的估计
- `Vds_keepalive_target` 是故意留给线性恒流环的最小生存余量

这个逻辑的物理意义是:

- 把采样电阻本来就必须吃掉的那部分压降单独算进去
- 只把真正留给 `Q5` 的线性调节余量压到最小

## 8. 动态余量应该怎么变化

当前讨论后的方向是:

- 低功率时，`Vds_keepalive_target` 可以相对大一点
- 高功率时，`Vds_keepalive_target` 应该更小

这样做的目标是:

- 小电流时先保稳定
- 大电流时尽量减少 `Q5` 线性区压降
- 让高功率工作点尽量靠近“Q5 接近全开”的状态

注意，这里变小的是:

`Vds_keepalive_target`

不是简单地说:

`Vdrain_gnd_target` 一定越大电流越小

因为 `Vdrain_gnd_target` 里还叠加了 `Vsense_est`。

### 8.1 我给出的第一版估计值

下面这组数值不是最终定值，而是我建议你上板优先评估的第一版草案。

建议把:

- `Vds_keepalive_target_max` 先设在 `800mV`
- `Vds_keepalive_target_min` 先设在 `200mV`

也就是:

- 低亮度时给 `Q5` 留 `0.8V` 左右真余量
- 高亮度时把 `Q5 true Vds` 压到 `0.2V` 左右

这是一个比较激进、明显偏向“高功率尽量让 Q5 接近全开”的版本。

### 8.2 如果先按亮度命令分段，建议先试这张表

| `light_dac_target_permille` | 建议 `Vds_keepalive_target` |
| --------------------------- | ------------------------- |
| `0 ~ 150`                   | `800mV`                   |
| `151 ~ 350`                 | `650mV`                   |
| `351 ~ 600`                 | `500mV`                   |
| `601 ~ 850`                 | `350mV`                   |
| `851 ~ 1000`                | `200mV`                   |

这张表表达的就是:

- 越往高功率走，越主动把 `Q5` 推向“接近全开”
- 但先不直接压到 `0mV`

### 8.3 如果你觉得这组值太激进，保守回退版可以先用这组

| `light_dac_target_permille` | 保守版 `Vds_keepalive_target` |
| --------------------------- | -------------------------- |
| `0 ~ 150`                   | `1000mV`                   |
| `151 ~ 350`                 | `850mV`                    |
| `351 ~ 600`                 | `700mV`                    |
| `601 ~ 850`                 | `500mV`                    |
| `851 ~ 1000`                | `350mV`                    |

我个人判断:

- 如果你现在目标是尽快验证“高功率下 Q5 尽量少烧压差”这个方向，优先试激进版
- 如果你更担心高亮掉线、热机失稳、运放失去调节权，优先试保守版

## 9. 在当前项目里，第一版动态余量应该怎么落地

由于 MCU 还没有直接闭环采样 `Iled`，所以不要一上来写得太复杂。

建议分两阶段。

### 9.1 第一阶段: 先按亮度命令估算

先用:

`light_dac_target_permille`

作为“目标电流强弱”的代理量。

也就是先写成:

`Vds_keepalive_target = f(light_dac_target_permille)`

逻辑上做成:

- 低亮度: keep-alive margin 稍大
- 中亮度: keep-alive margin 中等
- 高亮度: keep-alive margin 更小

这样虽然还不是严格电流闭环，但方向是对的，而且实现成本低。

我建议你第一版就直接按上面的分段表来，不要一开始写连续函数。

原因是:

- 分段表更容易在 debug 里看出每一档到底发生了什么
- 如果某一档明显太激进，只改一个区间的值就能回退
- 你现在还没有足够电流实测数据，不适合先写得太数学化

### 9.1.1 一个可直接代入的线性版本

如果你后面不想用分段，而想先用简单直线，也可以先试这条:

`Vds_keepalive_target_mv = 850 - 0.65 * light_dac_target_permille`

再限幅到:

- 最小 `200mV`
- 最大 `800mV`

代几个点大概是:

- `50 permille -> 818mV`
- `200 permille -> 720mV`
- `500 permille -> 525mV`
- `800 permille -> 330mV`
- `1000 permille -> 200mV` after clamp

这条线的好处是实现简单，坏处是没有分段表直观。

### 9.2 第二阶段: 再按实测电流或拟合电流估算

后续如果把:

- `DAC_CUR_CTR`
- `R14`
- 实测 LED 电流

之间关系拟合出来，就可以改成:

`Vsense_est = Icmd_est * 0.1 ohm`

再进一步写成:

`Vdrain_gnd_target = Vsense_est + Vds_keepalive_target(Icmd_est)`

这才是更完整、更物理一致的版本。

### 9.2.1 我建议你后面优先试的电流版数值

如果你后面能得到一个粗略的 `Icmd_est`，我建议先试这组:

| `Icmd_est` | `Vsense_est` with `0.1 ohm` | 建议 `Vds_keepalive_target` | 得到的 `Vdrain_gnd_target` |
| ---------- | --------------------------- | ------------------------- | ----------------------- |
| `0.2A`     | `20mV`                      | `800mV`                   | `820mV`                 |
| `0.5A`     | `50mV`                      | `650mV`                   | `700mV`                 |
| `1.0A`     | `100mV`                     | `500mV`                   | `600mV`                 |
| `1.5A`     | `150mV`                     | `350mV`                   | `500mV`                 |
| `2.0A`     | `200mV`                     | `200mV`                   | `400mV`                 |

这组数值体现的是你当前想要的工程倾向:

- 电流越大，`Q5 true Vds` 留得越小
- 但 `Vdrain_gnd_target` 不会直接降到接近 `0`
- 这样既逼近全开，又保留一点调节权

如果你的满功率电流不是 `2A`，就按 `Vsense = I * 0.1 ohm` 重新换算即可。

## 10. 当前阶段绝对不要做的事

### 10.1 不要继续死控固定 `36.42V`

因为这会让多出来的压差长期烧在 `Q5` 上。

### 10.2 不要把 `Q5` 硬顶死当成正常目标

因为当前架构里 Boost 没有接管主电流环。

### 10.3 不要让 Boost 为了追余量无上限升压

如果出现下面这种情况:

- 亮度命令已经很高
- `VOUT` 已经逼近绝对上限
- 但目标余量仍然达不到

那就说明当前系统能力不足，或者当前目标过于激进，不能继续无脑升压。

这时后续应考虑:

- 进入“掉线/能力不足”判据
- 限制亮度命令
- 或提高 keep-alive target

而不是继续顶高 `VOUT`

### 10.4 我建议你后面加的几个数值边界

这些值也先给你一个草案，后面可以边测边收:

- `Vout_abs_max = 42000mV` 已有
- `Vout_soft_ceiling = 39000mV`
- `Vdrain_gnd_target_max = 1200mV`
- `Vdrain_gnd_target_min = 300mV`
- `Vds_keepalive_target_min = 200mV`
- `Vds_keepalive_target_max = 1000mV`

这些边界的目的不是让系统变保守，而是:

- 防止 ADC 抖动或异常估算把目标压得过低，导致恒流环直接死掉
- 防止控制目标被抬得过高，重新回到“Q5 白白吃大压差”的老路

### 10.5 我建议的“能力不足”判据草案

这个也不是最终代码，只是后续逻辑草案:

如果同时满足:

- `light_dac_target_permille >= 800`
- `vout_mv_lpf >= 40000mV`
- `vds_mv_lpf < vdrain_gnd_target - 150mV`
- 持续时间 `> 50ms`

就可以判成:

- 当前亮度命令已经逼近系统能力上限
- Boost 不能再继续无脑升压
- 后续应该限制亮度，或切换到保守余量表

## 11. 当前项目建议采用的策略顺序

1. 先保留现有固定 `1V` 版本用于继续观察。
2. 上板记录不同亮度下 `vds_mv_lpf`、`vout_mv_lpf`、`boost_duty_permille` 的行为。
3. 再把固定余量改成“随亮度命令变化的 keep-alive margin”，优先用上面的分段表。
4. 若高亮掉线，就先切回保守表，不要继续死顶。
5. 后续再把亮度命令拟合成 `Icmd_est`。
6. 最后再做“按估算电流计算 `Vdrain_gnd_target`”的版本。

## 12. 调试时最值得盯的量

当前代码或后续演进里，最有价值的 debug 量是:

- `vds_mv_lpf`
- `vout_mv_lpf`
- `vout_ctrl_ref_mv`
- `vds_target_mv`
- `boost_duty_permille`
- `light_dac_target_permille`
- `boost_voltage_debug.vout_minus_vds_mv`

如果后面加动态余量，建议再增加:

- `vds_keepalive_target_mv`
- `vsense_est_mv`
- `vdrain_target_mv`
- `headroom_mode`

## 13. 一句话结论

当前项目下一步该做的，不是让 `Q5` 彻底硬全开，而是:

在高功率下让 `Q5` 尽量接近全开，只保留最小必要调节余量。

这就意味着:

- 余量目标必须从固定值演进为动态值
- 动态目标应围绕 `Vsense_est + Vds_keepalive_target` 设计
- `Vds_keepalive_target` 的方向是“低功率略大，高功率更小”
- 但绝不能减到让模拟恒流环完全失去控制权
